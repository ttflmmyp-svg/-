# 《把今天洗一下》联网版 v0.1 技术方案

更新日期：2026-06-06  
负责人：B  
当前阶段：技术方案冻结，尚未开始正式编码

## 1. v0.1 目标

联网版 v0.1 的目标不是一次性做完所有功能，而是先做出一个完整、可运行、可接 API 的主流程版本。

用户应该能完成这条体验：

```text
首页
→ 洗衣房界面
→ 输入情绪动画（聚焦脏衣篓）
→ 投衣动画
→ 清洗页（聚焦洗衣机）
→ 小票页（情绪洗衣单）
```

v0.1 成功标准：

- 用户能从首页进入洗衣房。
- 用户能输入一段坏情绪文本。
- 系统能生成情绪分析结果。
- 用户能看到对应的脏衣服。
- 用户能完成一轮清洗互动。
- 用户能得到一张情绪小票。
- API 失败时，仍然可以用本地 mock / 本地规则走完整流程。

## 2. 产品边界

当前版本只做单次情绪清洗流程。

明确不做：

- 长期心理评估。
- 医疗诊断。
- 复杂社交系统。
- 登录注册。
- 云端历史记录。
- 多用户同步。
- 衣柜页。
- 我的小票页。

衣柜页和我的小票页属于可选功能，v0.1 不纳入必做范围。

## 3. 推荐技术栈

第一阶段是联网 API 完整版，推荐使用：

- 前端：Vite + React + TypeScript。
- 样式：CSS Modules 或普通 CSS，先不引入重型 UI 组件库。
- 后端：Node.js + Express。
- API 调用：前端请求本地后端，由后端转发大语言模型 API。
- 数据库：v0.1 暂不使用。
- 本地存储：可用 localStorage 保存最近一次小票，非必需。

选择原因：

- React 方便管理多阶段状态。
- TypeScript 方便约束 API 返回结构。
- Express 后端可以保护 API key，不把密钥写在前端。
- 不上数据库可以减少工程量。
- 后续裁剪离线版时，可以把核心规则和文案库迁移到纯前端版本。

## 4. 页面与状态机

冻结表里的页面更适合做成“洗衣房主场景 + 状态切换”，而不是很多独立页面。

建议使用一个主状态 `sceneStep` 控制体验进度。

```ts
type SceneStep =
  | "home"
  | "laundryRoom"
  | "basketInput"
  | "throwCloth"
  | "soap"
  | "drum"
  | "bubble"
  | "spin"
  | "hang"
  | "receipt";
```

### 4.1 首页 home

目的：

- 简单介绍作品。
- 不做复杂 landing page。
- 提供“开始洗今天”按钮。

主要元素：

- 作品名：《把今天洗一下》。
- 一句话介绍。
- 开始按钮。

进入下一步：

- 点击开始按钮 → `laundryRoom`。

### 4.2 洗衣房界面 laundryRoom

目的：

- 建立主场景。
- 展示洗衣机、脏衣篓、NPC、操作区。

主要元素：

- 洗衣机。
- 脏衣篓。
- NPC。
- 今日洗涤入口。

进入下一步：

- 点击脏衣篓或“放入今天” → `basketInput`。

### 4.3 输入情绪 basketInput

目的：

- 聚焦脏衣篓。
- 让用户输入今天想洗掉的情绪。

主要元素：

- 文本输入框。
- 提示文案。
- 确认投递按钮。
- API 分析 loading 状态。

行为：

- 保存 `userText`。
- 调用后端分析接口。
- API 成功后保存分析结果。
- API 失败时使用本地 fallback。

进入下一步：

- 分析完成 → `throwCloth`。

### 4.4 投衣动画 throwCloth

目的：

- 把用户输入转化成一件脏衣服。
- 展示从脏衣篓到洗衣机的投递动画。

主要元素：

- 情绪衣服。
- 洗衣机。
- 动画提示。

行为：

- 根据 `emotionType` 显示不同衣服样式。
- 播放投衣动画。

进入下一步：

- 动画结束或点击继续 → `soap`。

### 4.5 肥皂涂抹 soap

目的：

- 第一段清洗互动。

主要元素：

- 脏衣服。
- 肥皂。
- 污渍区域。

交互：

- 鼠标/手指拖动肥皂涂抹。
- 简化版可以用点击 3 次代替涂抹。

进入下一步：

- 涂抹进度达到目标 → `drum`。

### 4.6 滚筒 drum

目的：

- 模拟洗衣机滚筒清洗。

主要元素：

- 洗衣机滚筒。
- 旋转进度。

交互：

- 桌面端：拖动滚筒旋转。
- 移动端：可以支持摇晃手机。
- v0.1 保底方案：点击或拖动累计进度。

进入下一步：

- 滚筒进度达到目标 → `bubble`。

### 4.7 戳泡泡 bubble

目的：

- 把坏想法具象成泡泡并戳掉。

主要元素：

- 5 个泡泡。
- 每个泡泡一句坏想法。

交互：

- 点击泡泡后消失。
- 全部消失后显示清洗完成。

进入下一步：

- 泡泡全部消失 → `spin`。

### 4.8 脱水 spin

目的：

- 让用户通过长按完成甩干。

主要元素：

- 长按按钮。
- 进度条。
- 洗衣机动效。

交互：

- 长按累计进度。
- 松手可以暂停。
- v0.1 保底方案：按住 2 秒即可完成。

进入下一步：

- 脱水完成 → `hang`。

### 4.9 晾晒 hang

目的：

- 将洗好的情绪晾起来，形成收束感。

主要元素：

- 晾衣绳。
- 洗净后的衣服。

交互：

- 拖动衣服到晾衣绳。
- v0.1 保底方案：点击“晾起来”完成。

进入下一步：

- 晾晒完成 → `receipt`。

### 4.10 小票页 receipt

目的：

- 给用户一个温柔、可截图的情绪洗衣单。

主要元素：

- 情绪洗衣单。
- 衣物类型。
- 情绪类型。
- 污渍来源。
- 今日洗涤结果。
- 温和表达。
- 今日建议。
- 洗净指数。
- 再洗一件按钮。

进入下一步：

- 点击再洗一件 → 回到 `basketInput` 或 `laundryRoom`。

## 5. 核心数据结构

建议前端维护一个 `gameState`。

```ts
type EmotionType =
  | "anxiety"
  | "anger"
  | "tired"
  | "lost"
  | "wronged"
  | "overthinking"
  | "lonely"
  | "meltdown"
  | "default";

type WashStage = "soap" | "drum" | "bubble" | "spin" | "hang";

type EmotionAnalysis = {
  emotionType: EmotionType;
  emotionLabel: string;
  intensity: number;
  keywords: string[];
  dirtyThoughts: string[];
  realFeeling: string;
  gentleReframe: string;
};

type Receipt = {
  title: string;
  clothName: string;
  emotionLabel: string;
  stainSource: string;
  washMode: string;
  resultText: string;
  gentleReframe: string;
  suggestion: string;
  cleanIndex: number;
  createdAt: string;
};

type GameState = {
  userText: string;
  analysis: EmotionAnalysis | null;
  clothName: string;
  washMode: "今日洗涤";
  washProgress: Record<WashStage, number>;
  bubbles: string[];
  receipt: Receipt | null;
  apiStatus: "idle" | "loading" | "success" | "fallback" | "error";
};
```

## 6. API 设计草案

v0.1 只需要一个核心接口。

### 6.1 分析接口

```text
POST /api/analyze-emotion
```

请求体：

```json
{
  "text": "我今天真的很烦，感觉什么都做不好"
}
```

成功返回：

```json
{
  "emotionType": "overthinking",
  "emotionLabel": "内耗",
  "intensity": 72,
  "keywords": ["烦", "做不好"],
  "dirtyThoughts": [
    "我什么都做不好",
    "今天又被我搞砸了",
    "是不是我不够好"
  ],
  "realFeeling": "你可能不是做不好，而是今天已经消耗了很多力气。",
  "gentleReframe": "今天可以先不证明自己，先把一件小事放回原位。",
  "suggestion": "给自己 10 分钟，只完成一个很小的动作。",
  "receiptTitle": "今日内耗洗涤单"
}
```

前端收到后生成：

- 衣服名称。
- 泡泡内容。
- 小票内容。

如果 API 失败：

- 使用本地 `detectEmotion(text)`。
- 使用本地 `receiptTemplates`。
- `apiStatus` 设置为 `"fallback"`。

## 7. 本地 fallback 规则

联网版也必须保留本地规则。

原因：

- API 可能超时。
- 现场网络可能不稳定。
- 后续离线版需要复用。

本地规则包括：

- 关键词情绪识别。
- 情绪类型到衣服名称的映射。
- 情绪类型到泡泡文案的映射。
- 情绪类型到小票文案的映射。

示例：

```ts
const emotionKeywords = {
  anxiety: ["焦虑", "担心", "来不及", "害怕", "紧张"],
  anger: ["烦死", "生气", "气死", "讨厌", "火大"],
  tired: ["累", "疲惫", "不想动", "困", "撑不住"],
  lost: ["迷茫", "不知道", "没方向", "空", "混乱"],
  wronged: ["委屈", "没人懂", "冤", "难受"],
  overthinking: ["内耗", "想太多", "做不好", "怪我", "自责"],
  lonely: ["孤独", "没人陪", "一个人", "被丢下"],
  meltdown: ["崩溃", "疯了", "受不了", "炸了"]
};
```

## 8. v0.1 开发任务拆分

### 任务 1：工程骨架

- 创建前端项目。
- 创建后端项目。
- 配置开发启动命令。
- 配置环境变量示例。
- 写基础目录结构。

完成标准：

- 前端能打开。
- 后端 `/health` 能返回正常。
- 前端能请求后端测试接口。

### 任务 2：主状态机

- 建立 `sceneStep`。
- 建立 `gameState`。
- 写 `goToStep(step)`。
- 首页能进入洗衣房。

完成标准：

- 不接 API 也能在各步骤之间切换。

### 任务 3：输入与分析

- 输入用户文本。
- 请求 `/api/analyze-emotion`。
- 接入 mock 返回。
- 保存分析结果。
- API 失败时 fallback。

完成标准：

- 输入一句话后能得到情绪类型和泡泡文案。

### 任务 4：投衣与清洗互动

- 根据情绪显示衣服。
- 投衣动画。
- 肥皂互动。
- 滚筒互动。
- 戳泡泡互动。
- 脱水互动。
- 晾晒互动。

完成标准：

- 用户能从输入走到清洗完成。

### 任务 5：小票页

- 生成小票数据。
- 展示情绪洗衣单。
- 提供再洗一件。
- 可截图。

完成标准：

- 小票内容完整，文案不空。

### 任务 6：视觉与移动端

- 洗衣房主场景。
- 基础动画。
- 手机端适配。
- 点击区域足够大。
- 页面不横向溢出。

完成标准：

- 手机和桌面都能完整玩一遍。

## 10. 关键约束

### 第一阶段联网版

- 可以联网。
- 可以接 API。
- 可以使用开发框架。
- 必须保护 API key。
- 必须有本地 fallback。
- 不做登录和数据库。
- 不做可选页面。

### 第二阶段离线版预留

从 v0.1 开始就要注意：

- 情绪规则独立成文件。
- 小票模板独立成文件。
- 不把核心流程完全绑死在 API 上。
- 素材命名保持稳定。
- 避免一开始就依赖重型动画库。

## 11. 暂定验收清单

v0.1 完成时，至少要满足：

- 首页能进入洗衣房。
- 用户能输入情绪。
- API mock 能返回分析结果。
- API 失败时 fallback 能接管。
- 情绪能映射成衣服。
- 清洗五阶段能走完。
- 小票能生成。
- 再洗一件能重新开始。
- 刷新页面不白屏。
- 手机端不横向溢出。
