## 1. 项目内容与目的

《把今天洗一下》是一个治愈向互动网页小游戏。

它把用户当天的坏情绪、烦恼或内耗文本，转化成一件“脏衣服”，再通过洗衣流程完成一次轻量、可玩的情绪整理。用户不是直接面对“我很糟糕”，而是面对“一件可以被洗一洗的心事”。

核心体验链路：

1. 用户进入情绪洗衣房。
2. 用户输入今天的烦恼或坏情绪。
3. 系统识别情绪类型，并生成对应的“脏衣服”。
4. 用户把衣服投入洗衣机。
5. 用户完成清洗互动。
6. 系统生成一张“情绪小票”。
7. 用户可以截图、保存或再洗一件。

项目最终希望呈现的是一个轻松、温柔、有记忆点的互动体验，而不是严肃的心理评估工具。

## 2. 当前总体策略

项目分两个版本推进。

### 第一阶段：联网 API 完整版

先做一个可以联网、可以接 API 的完整项目。

这个版本不先受 8MB 体积和离线限制约束，重点是把完整体验做出来：

- 页面流程完整。
- 用户输入可以被 API 分析。
- 情绪识别、坏想法提取、温和表达重组、小票文案生成可以接入大语言模型。
- 视觉和交互可以比离线版更丰富。
- 作为理想体验版、开发母版和演示增强版。

### 第二阶段：8MB 离线提交版

在完整项目完成后，再裁剪出一个 8MB 以内、不能联网的小版本。

这个版本用于最终提交和现场兜底：

- 不依赖 API。
- 不使用 CDN。
- 不使用外部图片链接。
- 所有素材本地化。
- 保留核心单次情绪清洗流程。
- 必须断网可打开、可玩、不卡死。

当前情绪类型包括：

- 焦虑。
- 愤怒。
- 疲惫。
- 迷茫。
- 委屈。
- 内耗。
- 孤独。
- 崩溃 / 发疯。

第一阶段联网版：

- 优先使用大语言模型 API 对用户输入进行分析。
- 输出情绪类型、关键词、坏想法、真实感受、温和改写、小票文案。
- API 失败时必须能降级到本地关键词规则。

第二阶段离线版：

- 使用本地关键词规则。
- 使用本地小票文案库。
- 不发起网络请求。

### 4.5 小票生成

小票需要体现本次清洗结果。

建议字段：

- 衣物类型。
- 情绪类型。
- 污渍来源。
- 洗涤模式。
- 清洗过程摘要。
- 温和表达。
- 今日建议。
- 洗净指数。
- 生成时间。

注意：洗净指数不建议写 100，应保留一点真实感，例如 30-88。

### 4.6 API 接入

第一阶段完整项目允许联网并接 API。

API 可能负责：

- 情绪识别。

- 真实感受拆解。
- 温和表达重组。
- 小票文案生成。
- NPC 台词生成或改写。

B 需要负责：

- 设计前端调用方式。
- 管理 loading 状态。
- 处理 API 失败。
- 避免页面因为接口错误白屏。
- 保留本地 fallback。
- 不把真实 API key 写死在前端公开代码里。

## 7. 约束条件

### 产品约束

- 作品是治愈向互动小游戏，不是医疗产品。
- 不做心理诊断。
- 不给严肃医疗建议。
- 文案要温和，不要评判用户。
- 用户输入可能很负面，结果要承接情绪，但不能放大伤害。

### 第一阶段联网版约束

- 可以联网。
- 可以接 API。
- 可以使用更完整的工程结构。
- 可以使用开发依赖和构建工具。
- 必须有 API 失败后的 fallback。
- 不应把敏感密钥暴露在前端代码中。

### 第二阶段离线版约束

- 总体积不超过 8MB。
- 不能联网。
- 不能依赖 CDN。
- 不能使用外部图片链接。
- 不能 fetch 远程接口。
- 必须断网能打开。
- 必须保留核心流程。
- 优先稳定，不追求复杂技术效果。

### B 开发约束

- 先做流程，再做美化。
- 先做可玩，再做高级交互。
- C 的素材没到时，用临时素材占位。
- A 的规则没完全定时，先用可替换的数据结构。
- 每个页面都要有下一步或返回方式，不能让用户卡死。
- 移动端按钮必须能点。
- 页面不能横向溢出。
- 刷新页面不能白屏。
- 最后 30 分钟不加功能，只修 bug、错字和提交材料。

### 手机端 playable demo 展示约束

当前阶段不是做正式 App，而是先用 HTML/CSS/JS 做一个能在手机浏览器里展示和测试的 playable demo。后续开发必须优先保证手机竖屏体验，不要按桌面网页思路开发。

#### 1. 主要展示场景

- 必须以手机竖屏为主要展示场景。
- 默认目标屏幕可参考 390px × 844px，但不能写死为唯一尺寸。
- 页面在不同手机宽度下不能横向溢出。
- 不能出现需要用户左右滑动的情况。
- 不要为了桌面端展示牺牲手机端效果。

#### 2. viewport 设置

HTML 的 `head` 中必须使用：

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover">
```

目的：

- 确保手机浏览器不会按桌面网页缩放页面。
- 支持 iPhone 刘海、灵动岛、底部横条等安全区处理。

#### 3. 页面基础尺寸

`html` 和 `body` 不能保留默认 margin / padding。

推荐基础约束：

```css
html,
body {
  margin: 0;
  padding: 0;
  width: 100%;
  height: 100%;
  overflow: hidden;
  background: #000;
  user-select: none;
  touch-action: manipulation;
}
```

要求：

- 主容器使用 `100vw` 和 `100dvh`。
- 优先使用 `100dvh` 处理手机浏览器地址栏伸缩问题。
- 页面整体不应出现普通网页滚动。
- 只有某个明确需要滚动的局部区域才允许内部滚动。

#### 4. 游戏主画面容器

所有游戏 UI、角色、按钮、背景都必须放在稳定的手机画布容器内。

推荐结构：

```css
.phone-page {
  width: 100vw;
  height: 100dvh;
  display: flex;
  justify-content: center;
  align-items: center;
  background: #000;
}

.game-screen {
  position: relative;
  width: min(100vw, 480px);
  height: 100dvh;
  overflow: hidden;
  background: #111;
  box-sizing: border-box;
  padding-top: env(safe-area-inset-top);
  padding-bottom: env(safe-area-inset-bottom);
}
```

要求：

- `.phone-page` 负责居中。
- `.game-screen` 承载所有 demo 内容。
- `.game-screen` 不允许被内部图片或元素撑开。
- 背景、按钮、角色、交互层都在 `.game-screen` 内定位。

#### 5. 定位方式

不能大量使用写死 px 定位。

避免：

```css
left: 320px;
top: 600px;
```

推荐：

```css
left: 50%;
bottom: 18%;
transform: translateX(-50%);
```

要求：

- 元素位置尽量使用百分比、`vw`、`vh`、`dvh`、`clamp()`。
- 尺寸可使用 `width: 28vw; max-width: 140px;` 这种响应式写法。
- 关键元素必须在不同手机尺寸下保持合理比例。

#### 6. 字体和元素尺寸

文字和控件必须适配手机。

推荐：

```css
font-size: clamp(16px, 4vw, 24px);
```

要求：

- 标题、按钮、提示文字不要写死过大或过小。
- 重要文字必须在手机上清晰可读。
- 文案不能溢出按钮或遮挡主要内容。

#### 7. 点击区域

所有可点击元素必须适合手指点击。

推荐：

```css
button {
  min-width: 44px;
  min-height: 44px;
  font-size: 16px;
}
```

要求：

- 最小点击区域不低于 44px × 44px。
- 按钮之间要有足够间距。
- 不要设计只能靠鼠标精确点击的交互。
- 复杂拖拽必须有点击或长按等保底方案。

#### 8. 交互反馈

不要依赖 hover 作为核心交互。

要求：

- 手机端核心反馈必须依赖 click / touch / active。
- 可以保留 hover 作为桌面预览效果，但不能作为必要路径。
- 点击反馈建议使用 `:active`。

示例：

```css
.button:active {
  transform: scale(0.96);
}
```

#### 9. 手机安全区

顶部和底部 UI 不能被 iPhone 刘海、灵动岛、底部横条遮挡。

要求：

- 顶部区域使用 `env(safe-area-inset-top)`。
- 底部按钮使用 `env(safe-area-inset-bottom)`。

示例：

```css
.bottom-controls {
  bottom: calc(20px + env(safe-area-inset-bottom));
}
```

#### 10. 禁止无意义缩放、选中、高亮

为了更像手机游戏体验，必须加入：

```css
* {
  -webkit-tap-highlight-color: transparent;
}

html,
body {
  user-select: none;
  touch-action: manipulation;
}
```

要求：

- 用户点击按钮时不要出现系统默认高亮。
- 页面内容不要被误选中。
- 不需要用户手动缩放页面。

#### 11. 图片和背景

图片和背景必须响应式。

要求：

- 背景图根据用途选择 `object-fit: cover` 或 `object-fit: contain`。
- 需要完整保留的图片使用 `contain` 或按原始比例适配。
- 不要让图片超出 `.game-screen` 导致页面撑开。
- 背景和交互按钮必须在同一个稳定容器内对齐。

#### 12. 手机浏览器兼容

开发完成后需要能在手机浏览器中直接打开。

目标环境：

- iPhone Safari。
- Android Chrome。
- 微信内置浏览器尽量兼容。

注意：

- Chrome DevTools 手机模拟器只能作为初步检查。
- 不能只按桌面浏览器窗口判断最终效果。

#### 13. 当前阶段验收标准

当前 playable demo 至少满足：

- 手机端打开后不横向溢出。
- 页面不会被浏览器地址栏、刘海、底部横条明显遮挡。
- 主要内容居中且比例合理。
- 按钮能正常点击，点击区域足够大。
- 不需要用户手动缩放页面。
- 不出现桌面网页式布局。
- 游戏 demo 看起来像一个手机竖屏画面，而不是普通网页。

## 8. 暂定风险

### API 风险

API 可能失败、超时、限流或返回格式不稳定。

应对：

- 前端必须有 loading、error 和 fallback。
- API 返回结构要尽量简单。
- 本地关键词判断不能删除。

### 工程量风险

完整联网版可能比预期大。

应对：

- 第一版只保留主流程。
- 可选页面后置。
- 复杂互动后置。

### 体积风险

离线版需要小于 8MB。

应对：

- 不放大图。
- 压缩素材。
- 少用重型库。
- 必要时用 CSS 和 emoji 替代图片。

### 现场演示风险

网络、浏览器、设备都可能出问题。

应对：

- 联网版用于展示增强体验。
- 离线版用于兜底。
- 保留一条最短演示路径。
